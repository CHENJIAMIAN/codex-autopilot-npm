# 自动跑轮次状态栏设计

## 背景

`codex-autopilot-npm` 当前是顺序执行型 Windows CLI。自动运行阶段已经会输出每轮开始/结束横幅、最后消息、窗口标题，但缺少一个持续可见的运行状态摘要。

这次改动的目标不是把 CLI 改成 TUI，而是在保持现有交互模型的前提下，为“自动跑轮次”阶段增加轻量、实时的状态栏。

## 目标

- 仅在自动跑轮次阶段显示状态栏
- 保持现有 CLI 结构，不引入 TUI 框架或重型依赖
- 在 PowerShell 终端中稳定工作
- 与现有日志、横幅、最后消息输出共存
- 代码改动集中在运行主循环，便于测试

## 非目标

- 不改造会话选择、提示语选择等交互步骤
- 不实现多面板、快捷键、实时日志窗口等 TUI 能力
- 不实现每秒倒计时刷新
- 不引入额外 npm 依赖

## 方案选择

### 方案 1：单行实时状态栏

在自动运行期间占用终端当前一行，使用回车符覆盖刷新，状态变化时重绘当前行。

优点：

- 最接近“状态栏”的预期
- 不需要整体切换到 TUI
- 运行时信息连续可见

缺点：

- 需要处理普通输出前后的状态栏清除与恢复

### 方案 2：阶段性状态日志

每到开始、重试、休眠、结束等阶段额外打印一条状态日志。

优点：

- 实现最简单
- 兼容性最好

缺点：

- 更像日志，不像状态栏

### 方案 3：仅增强窗口标题

继续依赖窗口标题表达状态，不在终端正文中新增常驻信息。

优点：

- 改动最小

缺点：

- 信息承载有限
- 用户可见性弱

## 选定方案

选择方案 1：单行实时状态栏。

理由：

- 能在不引入 TUI 的情况下，显著提升运行阶段的可观测性
- 现有输出点集中在 `src/autopilot.js`，适合封装成小型渲染器
- 相比阶段性打印，更能满足“状态栏”这个明确诉求

## 显示内容

状态栏默认显示以下字段：

- 阶段：`准备中`、`运行中`、`重试中`、`休眠中`、`已完成`、`失败`
- 当前轮次：`Turn x/y`
- 重试信息：`retry a/b`
- 会话摘要：`session <前 8 位>`
- 工作目录摘要：`cwd <basename>`
- 休眠阶段附加 `sleep Ns`

示例：

`[运行中] Turn 3/10 | retry 0/2 | session abcdef12 | cwd codex-autopilot-npm`

`[休眠中] Turn 3/10 | retry 0/2 | session abcdef12 | cwd codex-autopilot-npm | sleep 3s`

## 渲染策略

- 状态栏只在主循环启动后启用
- 用 `\r` 回到行首，通过空白填充清除残留字符
- 当需要输出横幅、最后消息、错误信息等普通文本时：
  - 先清掉状态栏
  - 输出普通文本
  - 若流程仍在运行，再恢复状态栏
- 结束或失败时，收尾打印换行，避免光标停留在状态栏行末

## 结构设计

新增一个轻量状态栏模块，例如 `src/status-line.js`，负责：

- 根据状态对象格式化文本
- 管理上一帧长度，避免短文本覆盖长文本后残留
- 提供 `render()`、`clear()`、`println()`、`finish()` 这类最小接口

`src/autopilot.js` 负责：

- 在自动运行阶段创建状态栏实例
- 在轮次开始、重试、读取消息、休眠、结束、失败等节点更新状态
- 将现有 `writeHost` 输出改为通过状态栏包裹，确保状态栏与普通输出互不污染

## 测试策略

优先补 `tests/autopilot.test.js`，覆盖：

- 进入轮次时会渲染状态栏
- 输出普通消息时会先清理状态栏再写入文本
- 失败和完成时会正确收尾
- 休眠阶段会显示对应阶段文本

测试方式保持当前风格，通过注入假的 `writeHost` 收集输出，不依赖真实终端。

## 风险与处理

- Windows 终端 ANSI 支持差异：本方案只依赖 `\r` 与空格清屏，避免复杂控制序列
- 普通输出覆盖状态栏：通过统一的包裹输出接口处理
- 文本过长：先不做列宽裁剪，后续如有需要再补

## 验收标准

- 自动跑轮次期间，终端可见实时状态栏
- 现有横幅、最后消息和错误信息仍然可读
- 不引入新依赖
- 自动化测试覆盖新增行为
